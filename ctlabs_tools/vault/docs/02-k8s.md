# 📖 `vault-k8s` Dynamic RBAC Examples

When you use `vault-k8s role create`, Vault acts as a Just-In-Time (JIT) Kubernetes administrator. When a user requests access, Vault dynamically creates a temporary ServiceAccount, generates a Role based on your YAML rules, and binds them together. 

When the user's Vault lease expires (e.g., after 1 hour), Vault automatically deletes the ServiceAccount and Role from the cluster, leaving zero standing privileges.

Here are the most common YAML rule templates you can use with the `--rules` flag.

---

## 🔍 Example 1: The "Read-Only Debugger"
**Use Case:** A developer needs to check on a production app, view logs, and see the status of deployments, but should **not** be able to change anything or read Kubernetes Secrets.

**File:** `debug-readonly.yml`
```yaml
rules:
  # Core resources: Pods, Logs, and Events
  - apiGroups: [""]
    resources: ["pods", "pods/log", "events", "configmaps"]
    verbs: ["get", "list", "watch"]
  
  # Apps resources: Deployments, ReplicaSets, StatefulSets
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets", "statefulsets", "daemonsets"]
    verbs: ["get", "list", "watch"]
    
  # Network resources: Services, Ingresses
  - apiGroups: ["networking.k8s.io"]
    resources: ["ingresses"]
    verbs: ["get", "list", "watch"]
```

**Creation Command:**
```bash
vault-k8s role create prod-cluster debug-role --rules debug-readonly.yml --namespaces "backend,frontend"
```

---

## 🚀 Example 2: The "CI/CD Pipeline Deployer"
**Use Case:** Your GitHub Actions or GitLab CI pipeline needs to deploy a new version of an app. It needs permission to create/update Deployments, Services, and ConfigMaps, but shouldn't have full admin rights.

**File:** `cicd-deploy.yml`
```yaml
rules:
  # Allow managing Deployments
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
    
  # Allow managing Services and ConfigMaps
  - apiGroups: [""]
    resources: ["services", "configmaps"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]

  # Allow restarting pods (modifying pod templates triggers this)
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "delete"]
```

**Creation Command:**
```bash
vault-k8s role create prod-cluster deploy-role --rules cicd-deploy.yml --namespaces "payment-api"
```

---

## 🚨 Example 3: The "Break-Glass Admin"
**Use Case:** An incident has occurred, and an SRE needs absolute control over a specific namespace to debug, port-forward, exec into containers, and modify secrets.

**File:** `namespace-admin.yml`
```yaml
rules:
  # The wildcard '*' grants absolute power over ALL resources in the targeted namespace
  - apiGroups: ["*"]
    resources: ["*"]
    verbs: ["*"]
```

**Creation Command:**
```bash
vault-k8s role create prod-cluster break-glass-role --rules namespace-admin.yml --namespaces "prod-backend"
```

---

## 🧠 How the Vault Kubernetes Secrets Engine Maps This
Under the hood, `vault-k8s` takes your YAML array and injects it directly into the `generated_role_rules` parameter of the Vault API.

If you specify `--namespaces "backend"`, Vault creates a localized `Role` and `RoleBinding` in that specific namespace. If you leave the namespaces blank (or configure it for cluster-wide access), Vault will generate a K8s `ClusterRole` and `ClusterRoleBinding` instead.



### Best Practices for JIT K8s Access:
1. **Never grant `secrets` access** unless absolutely necessary. (Developers should be reading secrets from Vault, not K8s!).
2. **Keep the TTL short.** For human access, a 1-hour TTL is standard. For CI/CD, a 15-minute TTL is usually plenty.
3. **Target specific namespaces.** Avoid cluster-wide roles unless you are building an explicit `cluster-admin` break-glass roleset.
