# 🛡️ Vault & Kubernetes: The Architectural Guide to Auth

This guide covers the three ways to link Vault to Kubernetes. Each has different security implications and setup requirements.

---

## 🏗️ The Core Components
Regardless of the setup, you always need:
1. **Vault Host:** `192.168.30.11`
2. **K8s API Host:** `192.168.30.31`
3. **CA Certificate:** Extracted via `kubectl config view --raw --minify -o jsonpath='{.clusters[0].cluster.certificate-authority-data}' | base64 -d > k8s_ca.crt`

---

## 📑 Setup Options Comparison

| Feature | 1. The "Reviewer-less" (Your Setup) | 2. The "Dedicated Reviewer" (Best Practice) | 3. The "In-Cluster" (Default) |
| :--- | :--- | :--- | :--- |
| **Vault Location** | Outside Cluster | Outside Cluster | Inside Cluster (as a Pod) |
| **Reviewer JWT** | None (Uses Pod's own JWT) | Manually created ServiceAccount Token | Automatic (Uses Vault's own SA) |
| **RBAC Requirement** | **Every** App SA needs `auth-delegator` | **Only Vault** SA needs `auth-delegator` | **Only Vault** SA needs `auth-delegator` |
| **Pros** | Simplest config (No tokens to manage) | Highest security; least privilege | Zero manual configuration |
| **Cons** | Higher RBAC overhead per app | Must manage/rotate the Reviewer token | Only works if Vault is in the cluster |

---

## 🛠️ Option 1: The "Reviewer-less" Setup (Direct Delegation)
*This is the mode where "Disable local CA/JWT" is ON but the Reviewer JWT field is empty.*



### 1. Vault Configuration
```bash
vault-auth k8s configure \
  --host "[https://192.168.30.31:6443](https://192.168.30.31:6443)" \
  --ca-cert k8s_ca.crt
# (Notice: No --reviewer-jwt provided)
```

### 2. Mandatory Kubernetes RBAC
Because Vault has no identity of its own, the **Application** must have permission to verify itself.
```bash
kubectl create clusterrolebinding app-delegator \
  --clusterrole=system:auth-delegator \
  --serviceaccount=prod:payment-api-sa
```

## 🛠️ Option 2: The "Dedicated Reviewer" Setup
*The most secure method for Vault instances running outside the cluster.*



### 1. Create a Long-lived Reviewer on K8s
```bash
# Create a dedicated SA for Vault
kubectl create serviceaccount vault-reviewer -n default
kubectl create clusterrolebinding vault-reviewer-binding \
  --clusterrole=system:auth-delegator \
  --serviceaccount=default:vault-reviewer

# Generate a long-lived token (Required for K8s 1.24+)
kubectl apply -f - <<EOF
apiVersion: v1
kind: Secret
metadata:
  name: vault-reviewer-token
  namespace: default
  annotations:
    kubernetes.io/service-account.name: vault-reviewer
type: kubernetes.io/service-account-token
EOF

REVIEW_TOKEN=$(kubectl get secret vault-reviewer-token -o jsonpath='{.data.token}' | base64 -d)
```

### 2. Vault Configuration
```bash
vault-auth k8s configure \
  --host "[https://192.168.30.31:6443](https://192.168.30.31:6443)" \
  --ca-cert k8s_ca.crt \
  --reviewer-jwt "$REVIEW_TOKEN"
```

## 🧪 Testing the Result (The Pod Login)
Once any of the above is configured, the pod login flow is identical:

```bash
# 1. Grab pod's local identity
K8S_TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)

# 2. Authenticate to Vault
curl -s -k -X POST \
  -d '{"jwt": "'"$K8S_TOKEN"'", "role": "payment-api-role"}' \
  [https://192.168.30.11:8200/v1/auth/kubernetes/login](https://192.168.30.11:8200/v1/auth/kubernetes/login) | jq
```

---

## 🎯 Summary: Which should you use?
1. **Use Option 1 (Reviewer-less)** if you want the fastest setup and don't mind adding a ClusterRoleBinding every time you create a new application namespace/service account.
2. **Use Option 2 (Dedicated Reviewer)** if you want to centralize permissions. Once the `vault-reviewer` is set up, you **never** have to touch K8s RBAC again when adding new apps to Vault.
